# Neo Link: интеграция продажи трафика

## 1. Схема работы

Neo Link позволяет подключить внешний Telegram-бот к системе продажи трафика.

Базовый поток:

1. Владелец бота открывает `Продажа трафика`.
2. Добавляет бота по токену.
3. Система валидирует токен через `getMe`.
4. Генерируется `API key`.
5. Заявка уходит на модерацию админу.
6. После одобрения внешний бот может использовать Neo Link как источник спонсоров.
7. При отписке пользователя от спонсора Neo Link автоматически списывает стоимость с баланса владельца бота.

## 2. Подключение бота

### Шаг 1. Добавление в Neo Link

В интерфейсе:

- `Продажа трафика`
- `➕ Добавить нового бота`
- способ подключения: `С токеном`
- отправить токен

После успешного добавления пользователь получает:

- API key бота
- статус `На модерации`
- доступ к настройкам

### Шаг 2. Модерация

После добавления:

- админу отправляется уведомление
- админ может `Одобрить` или `Отклонить`

Статусы:

- `🟡 На модерации`
- `🟢 Допущено`
- `🔴 Отклонено`

## 3. Настройки бота

Для подключённого бота доступны:

- актуализация токена
- отображение в спонсорах:
  - боты: да/нет
  - ресурсы: да/нет
- максимальное количество спонсоров: `1..10`
- время сброса списка
- показ новых спонсоров: `1..96` часов
- базовая цена за подписку
- анти-скам режим
- лимит подозрительных действий

## 4. API-взаимодействие

В этом проекте реализованы:

- хранение `bot_token`
- хранение `api_key`
- учёт событий подключённого бота
- учёт подписок по спонсорам
- аудит отписок
- автоматическое списание средств

Рекомендуемый HTTP-контракт для шлюза Neo Link:

### `POST /api/neolink/register-subscription`

Назначение:

- зарегистрировать, что пользователь подписался на спонсора

Пример JSON:

```json
{
  "api_key": "bot_api_key",
  "user_id": 123456789,
  "sponsor_chat_id": "-1001234567890",
  "sponsor_name": "Sponsor channel",
  "order_id": 77,
  "charge_amount": 1
}
```

### `POST /api/neolink/check-member`

Назначение:

- запросить проверку подписки перед выдачей награды или действия

Пример JSON:

```json
{
  "api_key": "bot_api_key",
  "user_id": 123456789,
  "sponsor_chat_id": "-1001234567890"
}
```

### `POST /api/neolink/report-suspicious`

Назначение:

- логировать подозрительную активность

Пример JSON:

```json
{
  "api_key": "bot_api_key",
  "user_id": 123456789,
  "reason": "too_many_checks",
  "meta": {
    "count": 42
  }
}
```

## 5. Логика оплаты

По текущей настройке сервиса:

- подписчик с проверкой: `2.5 ₽`
- переход/подписчик без проверки: `1.5 ₽`
- при отписке списание идёт по цене подписки, зафиксированной для бота

Как это работает в проекте:

1. Подписка регистрируется в `traffic_bot_subscriptions`.
2. Фоновая задача регулярно вызывает `getChatMember` через токен подключённого бота.
3. Если пользователь уже не состоит в спонсоре:
   - подписка переводится в `unsubscribed`
   - фиксируется событие
   - с баланса владельца бота списывается сумма
   - создаётся запись в `transactions`

## 6. Проверка подписки

Для проверки используется Telegram Bot API:

- `getChatMember`

Успешными считаются статусы:

- `member`
- `administrator`
- `creator`

Если статус другой или Telegram вернул ошибку доступа, подписка считается неподтверждённой.

## 7. Обработка ошибок

Основные причины сбоев:

- токен бота невалиден
- бот не состоит в канале-спонсоре
- бот не администратор канала
- Telegram API временно недоступен
- таймаут сети
- дублирующая регистрация одного и того же токена
- попытка изменить настройки чужого бота

Рекомендуемая стратегия:

- всегда оборачивать `getMe` и `getChatMember` в `try/except`
- логировать текст ошибки в таблицу логов
- не выполнять повторное списание, если запись уже помечена как `charged`
- показывать пользователю понятный текст, не раскрывая внутреннюю трассировку

## 8. Анти-скам рекомендации

Рекомендуется:

- ограничивать частоту проверок
- хранить журнал подозрительных действий
- блокировать повторные циклы быстрой перепроверки
- отделять модерацию новых ботов от рабочих ботов
- не принимать подписки без подтверждения через Telegram API
